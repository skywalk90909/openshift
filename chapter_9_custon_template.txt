In VMware, you create a golden template VM — pre-patched, pre-configured — and every new VM is cloned from it. Chapter 9 does the same in OpenShift using three building blocks: Boot Source (the OS disk), Instance Type (CPU/RAM sizing), and Template (the complete VM blueprint).

Think of it like ordering a laptop:

Boot Source = the OS image (Windows 11 ISO)

Instance Type = the hardware spec (8 core / 16GB)

Template = the pre-configured standard laptop your IT team issues to everyone

9.1 & 9.2 — Creating and Saving Custom Boot Sources
Boot Source = a PVC containing a ready-to-boot OS disk image. This is the foundation — without it, you can't create VMs from templates.

Four ways to create a Boot Source:

text
Method 1 — Upload local file
  Your machine → qcow2/ISO file → HTTP upload → PVC created
       (good for air-gapped environments)

Method 2 — From URL
  OpenShift pulls from → https://myserver/rhel9.qcow2 → PVC created
       (CDI DataVolume does the import automatically)

Method 3 — Clone existing PVC
  Existing golden VM disk → CSI clone → new Boot Source PVC
       (fastest if you already have a patched disk)

Method 4 — From Container Registry
  OCI image in your registry → pulled → PVC created
       (GitOps-friendly — version control your OS images)
Where Boot Sources live:

text
namespace: openshift-virtualization-os-images
  ├── rhel9-boot-source  (PVC)    ← Red Hat provided, auto-updated
  ├── win2022-boot-source (PVC)   ← your custom upload
  └── ubuntu-boot-source  (PVC)   ← your custom upload
Auto-updating Boot Sources with DataImportCron:

text
DataImportCron (runs weekly)
     │
     ▼
Polls Red Hat CDN / your registry for newer OS image
     │
     ▼
New PVC created with updated image
     │
     ▼
DataSource pointer updated → all new VMs from template automatically
get the latest OS version — no manual intervention
DataImportCron = like your Ansible playbook that checks for new RHEL cloud images weekly and updates your Packer template automatically

9.3 & 9.4 — Instance Types and Templates
Three layers that combine to create a VM:

text
Boot Source (OS disk)
      +
Instance Type (hardware sizing)
      +
Preference (OS-specific tuning)
      =
VirtualMachine (running VM)
Instance Types — Define CPU/RAM Profiles
Instead of hardcoding cpu: 4, memory: 8Gi in every VM YAML, you define reusable instance type objects:

text
VirtualMachineClusterInstancetype  ← cluster-wide (admin defines)
  ├── u1.micro   →  1 CPU,  1Gi RAM
  ├── u1.small   →  1 CPU,  2Gi RAM
  ├── u1.medium  →  1 CPU,  4Gi RAM
  ├── u1.large   →  2 CPU,  8Gi RAM
  └── u1.xlarge  →  4 CPU, 16Gi RAM

VirtualMachineInstancetype  ← namespace-scoped (team defines their own)
  └── my-app-size → 8 CPU, 32Gi RAM
Creating a custom instance type:

bash
# Using virtctl (easiest)
virtctl create instancetype \
  --name my-app-size \
  --cpu 8 \
  --memory 32Gi | oc apply -f -
Preferences — OS-Specific Defaults
Preferences define things like: "For Windows VMs, use UEFI firmware, virtio drivers, and RTC clock." You don't hardcode this per-VM — you reference a preference object:

text
VirtualMachineClusterPreference
  ├── rhel.9         → BIOS: UEFI, CPU model: host-passthrough, disk bus: virtio
  ├── windows.2k22   → BIOS: UEFI, TPM enabled, disk bus: sata, clock: utc
  └── ubuntu.22.04   → BIOS: BIOS, disk bus: virtio
VM Template — The Full Blueprint
A Template ties Boot Source + Instance Type + Preference together:

text
Template: "RHEL 9 Standard App Server"
  ├── Boot Source  → rhel9-boot-source PVC (cloned per new VM)
  ├── Instance Type → u1.large (2 CPU, 8Gi RAM)
  ├── Preference   → rhel.9 (UEFI, virtio disk)
  └── Extra config → SSH key injected, cloud-init for hostname
End-to-End Flow: Developer Creates a VM from Custom Template
text
Step 1: Admin prepares once
        ─────────────────────
        Upload RHEL9 qcow2 → Boot Source PVC
        Create InstanceType u1.large (2CPU/8Gi)
        Create Preference rhel.9
        Publish Template "rhel9-app-server"
              │
              ▼
Step 2: Developer creates VM (30 seconds)
        ─────────────────────────────────
        oc create vm my-app-vm \
          --instancetype u1.large \
          --preference rhel.9 \
          --volume-datasource rhel9-boot-source
              │
              ▼
Step 3: What happens under the hood
        ────────────────────────────
        Boot Source PVC  ──CSI Clone──▶  new PVC (my-app-vm-disk)
              +
        InstanceType (2CPU/8Gi) applied to VM spec
              +
        Preference (UEFI/virtio) applied to VM spec
              │
              ▼
        VirtualMachine created → VM boots in ~30 seconds
Two Scopes of Instance Types — When to Use Which
Type	Scope	Who Creates	Use Case
VirtualMachineClusterInstancetype	Entire cluster	Cluster admin	Standard sizing for all teams (like AWS t3.large)
VirtualMachineInstancetype	Single namespace	Team/developer	Custom sizing for specific app needs
VirtualMachineClusterPreference	Entire cluster	Cluster admin	OS defaults (Windows/RHEL/Ubuntu)
VirtualMachinePreference	Single namespace	Team/developer	Custom OS tuning per namespace
